### 🎯 Task 7: Configure Storybook

---

#### 🧠 Engineering Context & Rationale

##### What Storybook Actually Solves — Not What The Marketing Page Says

Storybook is sold as "an isolated component development environment." That description is accurate but undersells what it does for a design system. The real value is a **forcing function**: a story file is a public commitment that a component renders correctly _without_ the props, context, providers, routes, or data it normally relies on inside the app.

For a portfolio that lives or dies by token discipline, this is non-negotiable. Without Storybook, a developer can write a `WindowChrome` component, render it inside an `/desktop` page that already has the Aero wallpaper, the Redux session, the Apollo client, and a focused window context — and be unable to tell which of those dependencies the component is actually using vs. accidentally inheriting. With Storybook, every story renders the component in a sandboxed iframe with **only** what the story explicitly provides. If your `WindowChrome` story breaks because it can't find a Redux store, you have just discovered an unintended coupling — before code review, before integration, before the bug ships.

##### Why Task 7 Comes Now (Not Later)

The phase order matters. Tokens (Task 6) had to land first because Stories will render _against_ the token system — Storybook's preview iframe loads `globals.css`, and every state variant of every primitive verifies that the tokens compose correctly. Tasks 8–9 (login screen primitives, login page) will be built **inside Storybook** before being assembled in `/login`. By Task 8, Storybook MUST already be running, configured, and stable.

Configuring Storybook before any primitives exist might feel premature — there's nothing to make a story for yet. The smoke-test story you'll write in this task verifies the **pipeline**, not a real component. Once the pipeline is proven, Task 8 can focus exclusively on component design without re-litigating Storybook plumbing under deadline pressure.

##### The Next.js Framework Adapter — The One Thing That Trips Everyone Up

Storybook is framework-agnostic at its core, but renders inside an iframe that has its own bundler config, its own webpack/vite pipeline, and its own module resolution rules. A vanilla Storybook install in a Next.js project will _almost_ work — you'll see styles render, but `next/image`, `next/font`, App Router context, the `@/`\* path alias, server component boundaries, and a half-dozen other Next-specific behaviors silently break or behave unexpectedly.

The fix is the `**@storybook/nextjs` framework adapter\*\* (selected automatically by the init script when it detects a Next project). This adapter:

- Mocks `next/router` and `next/navigation` so components that use `useRouter` / `usePathname` don't crash
- Wires up the same webpack config Next.js uses, so module resolution matches production
- Handles `next/image` so `<Image>` renders without a Next dev server backing it
- Respects `next.config.ts` redirects, image domains, and CSS handling
- Provides MDX docs that render with the project's actual styling

If you find yourself adding webpack overrides to `.storybook/main.ts` to fix something framework-y, **stop** — the adapter probably already handles it, and you're patching a symptom.

> **AGENTS.md reminder:** "This is NOT the Next.js you know." Before configuring Storybook, read `node_modules/next/dist/docs/` for any breaking changes the Storybook adapter must accommodate. The init may complete successfully but produce a config that needs tweaking for this Next version.

##### The Three Files You Will Touch

```
.storybook/
  main.ts        ← framework selection, addons, story file glob
  preview.ts     ← global decorators, parameters, globals.css import
  ↑
  Generated by `npx storybook@latest init` — you EDIT them, you don't write them from scratch.

src/
  components/
    YourFirstComponent/
      YourFirstComponent.tsx
      YourFirstComponent.stories.tsx   ← co-located with the component, NOT in a separate /stories/ folder
```

The co-location convention matters: every component lives in its own folder with its stories. This keeps the design system browseable — find the component file, you've found its stories. Centralized `/stories/` directories rot quickly because story file location decouples from component file location.

##### Component Story Format 3 (CSF3) — Type-Safe Stories

Stories are written in **CSF3**, the current Storybook story format. The shape:

```
const meta = { ... }     ← metadata: title, component, parameters, decorators
export default meta

export const Default = { ... }   ← named export = one story
export const Focused = { ... }
```

Each named export is one story, typed with `StoryObj<typeof meta>` — this gives you autocomplete on `args`, `parameters`, `decorators`, and the component's own props. Older formats (CSF1/2) are still supported but deprecated. **Always write CSF3.**

##### What You Are NOT Doing In This Task

Resist the urge to:

- Add a `ReduxProviderWrapper` global decorator. There are no Redux-dependent components yet. Adding the decorator preemptively introduces an unintended coupling: every story re-renders on every Redux dispatch even if it doesn't care, and your sandboxing claim weakens. Add the decorator **per-story** when an actual Redux-consuming component arrives in Task 8 or beyond.
- Add an Apollo decorator for the same reason.
- Build real component stories. The smoke-test story exists only to prove that the pipeline works end-to-end. Real primitives are Task 8.
- Configure Chromatic, visual regression testing, or CI integration. Out of scope. Phase 4.

The minimum sufficient scope is: Storybook installs, `globals.css` is loaded into the preview iframe via a single global decorator, one trivial story renders correctly using semantic tokens, and `npm run storybook` opens a working sidebar at `localhost:6006`.

  parameters: {

    //   1. A "Light" background — white or near-white, used for stories on opaque
    //      surfaces (window bodies, content areas)

    //      mimics the desktop, used for testing glass surfaces with backdrop-filter

    // background, your glass tokens will appear invisible in stories. The default
    // background should be the wallpaper one so your default story looks correct.
    backgrounds: {
      /* fill in */
    },

    controls: {
      matchers: {
        color: /(background|color)$/i,
        date: /Date$/,
      },
    },
  },
}

> **Why backgrounds, specifically:** This is the most non-obvious Storybook configuration step for a glass design system. Stories render in an isolated iframe with NO wallpaper, NO body background image, nothing. A glass component (background `rgba(255,255,255,0.15)` + `backdrop-filter: blur(10px)`) renders against pure white default and looks broken because there's nothing to blur. Configuring an Aero-themed background fixes this without forcing every story to wrap itself in a wallpaper div.

**1.** Storybook is sold as "isolated component development." But isolation has a specific _cost_ the marketing page does not mention. Name one concrete category of bug Storybook will _fail to catch_ that integration tests or the running app would catch — and explain why isolation is the direct cause of that blind spot. Then state how this project's planned testing stack (Jest + RTL + Cypress) covers that gap.

```
[Answer]:
Cross-component state contract bugs. Concrete example for this project: WindowChrome
will dispatch `focusWindow({ id })` when clicked, and Taskbar will read
`state.windows.focusedId` to highlight the corresponding taskbar entry. The two
components share a contract that lives entirely in Redux — action name, payload
shape, selector key. If someone renames `focusedId` to `activeWindowId` and updates
WindowChrome but forgets Taskbar, every Storybook story for both components still
renders correctly because each story mounts its own throwaway store (or none at
all) and never exercises the link between them. Isolation IS the direct cause:
the story explicitly does NOT mount both components in the same provider tree,
so the integration surface where the contract actually lives is never touched.

How the rest of the stack covers this:
  - Jest + RTL — write an integration test that renders <Taskbar /> and
    <WindowChrome /> inside the SAME real <Provider store={makeStore()}>, fires
    a click on the window, and asserts the taskbar item gains the focused class.
    The renamed selector breaks this test immediately.
  - Cypress — the E2E desktop suite clicks a real window in a real browser and
    asserts the real taskbar updates. This catches the same class of bug
    end-to-end and also catches the variant where the contract is correct but a
    CSS specificity change hides the focused-state styling.

Storybook proves each component is internally correct; RTL proves their contracts
compose; Cypress proves the composition survives the full runtime.
```

**2.** You configured an "Aero Wallpaper" background in `preview.ts` and set it as the default for the `GlassSurface` story. A teammate proposes "let's just put a wallpaper div inside every glass story — that way each story is self-contained and we don't need a Storybook-specific config." Identify two concrete failure modes that proposal introduces specifically because of how `backdrop-filter` interacts with stacking contexts and parent compositing.

```
[Answer]:
1. Stacking context capture breaks the backdrop sample.
   `backdrop-filter` samples everything painted BEHIND the element in the same
   stacking context — but only up to the nearest ancestor that creates a NEW
   stacking context. The moment a story author wraps the glass in a
   <div class="wallpaper"> and that wrapper (or any intermediate decorator) gets
   `transform`, `filter`, `opacity < 1`, `will-change`, `isolation: isolate`, or
   `position: fixed/sticky` applied — all of which Storybook's
   addon-a11y outlines, addon-viewport scaling, and decorators commonly do —
   the wrapper becomes the new stacking-context root and the backdrop the glass
   actually sees is just the wallpaper div's painted output, which the browser
   may flatten to a solid average color. Result: glass renders as a flat tinted
   rectangle with zero blur, and it looks "broken" inconsistently across stories
   depending on which addons are toggled. Hoisting the wallpaper to the iframe
   body via the backgrounds parameter puts it OUTSIDE every per-story
   stacking context, so the sample is always intact.

2. Parent compositing layer collapse hides the blur.
   When the wallpaper div is the immediate parent of the glass element, the
   browser's compositor frequently promotes them into the same compositing
   layer for performance — and `backdrop-filter` is specified to blur only
   ancestor content from BEHIND the element's compositing layer. Same-layer
   content is excluded. The visible failure: the glass gets the tint and the
   border but no blur — and worse, this varies between Chromium and Firefox
   because their layer-promotion heuristics differ, so the bug ships to one
   browser and not the other. Putting the wallpaper at the iframe body keeps
   it in the root compositing layer, guaranteeing the glass is always in a
   downstream layer that the filter can sample.

Both failures are silent in dev (story author sees a tint and assumes the
blur is "subtle") and only get caught in production when a screenshot diff
or a manual review notices the glass looks pasted-on rather than frosted.
```
